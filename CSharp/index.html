請求書・入金管理 WPF クライアント

Invoice Management System (Lite) のASP.NET Core Web APIを利用する、C# / WPF製の業務向けデスクトップクライアントです。
Web版と同じ請求・入金ドメインを題材に、JWT認証、Admin / Memberのロール制御、請求・入金・会員・売上・催促管理、PDF・CSV出力を実装しています。 WPF版では、表形式データの一覧性とデスクトップアプリらしい操作導線を意識して再構成しました。

GitHub リポジトリ 動画デモを見る Web版デモを見る Web版の詳細を見る

※WPFクライアント自体はデスクトップアプリのため、ブラウザ上の公開デモはありません。Web版デモでは共通する業務機能を確認できます。
※掲載データ・アカウント・請求情報はすべてダミーです。

実装状況

共通APIへ実際に接続し、管理者・会員それぞれの業務フローをWPF画面として実装しています。

実装済み（現時点）

  • JWTログイン／ログアウト、Admin / Memberロール別画面遷移
  • 会員ダッシュボード、プロフィール確認・編集
  • 会員用請求書一覧・詳細、PDF表示、未入金状況確認
  • 管理者ダッシュボード、会員一覧・詳細・退会（無効化）
  • 管理者用請求書一覧・詳細・作成・編集
  • 売上一覧、顧客別集計、月別売上グラフ
  • 入金一覧・詳細・手動登録・請求書への割当管理
  • 催促設定、文面プレビュー、催促履歴管理
  • 請求書PDF表示、売上一覧・集計結果のCSV出力
  • appsettings.jsonによるAPIベースURL管理

※画面モックではなく、既存のASP.NET Core Web APIからデータを取得・更新するクライアントとして実装しています。

今後の改善案

  • 共通スタイルとUIコンポーネントの整理・再利用性向上
  • 入力バリデーションとエラーメッセージの改善
  • APIエラー・業務エラー・認証切れの表示ルール統一
  • Service／Dialog／ファイル出力まわりの抽象化強化
  • 画面・ViewModelごとの責務分離の明確化
  • 境界値・異常系を含むテスト対象の継続拡充

Web版とのAPI互換性を維持しながら、デスクトップアプリとしての操作性・保守性・障害耐性を高める方針です。

設計方針

  • View：画面表示とユーザー操作の受け取り
  • ViewModel：入力値・画面状態・コマンド・画面遷移の管理
  • Service：HttpClientを利用したAPI通信
  • Model：DTO／Request／Response／表示用データの保持
  • Web版の単純移植ではなく、一覧性・視認性・役割別導線をWPF向けに再設計

品質・テスト

  • xUnitで19種類の主要ViewModelをテスト対象として整理
  • 検索条件・ページング・集計値・バリデーション・DTO組み立てを検証
  • IAuthServiceなどのinterfaceとFake Serviceで外部API依存を分離
  • エラー時のHasError／ErrorMessage、画面遷移Action・Commandを検証
  • GitHub ActionsのWindows環境でbuild／testを自動実行

UI自体ではなく、業務ロジックと画面状態を担うViewModelを中心に自動テストしています。

設計・共通APIとの連携

区分 主な責務・機能 WPFクライアントでの利用
View / ViewModel 表示・入力・コマンド・画面状態・画面遷移 DataGrid、検索条件、ページング、ローディング、エラー状態を管理
Service HttpClientによるAPI通信 認証、会員、請求、入金、売上、催促の通信処理を画面から分離
Auth API ログイン（/auth/login） JWTとユーザー情報を保持し、Admin / Member別の初期画面へ遷移
Invoices API 一覧・検索・詳細・作成・編集・PDF 請求明細、入金済み金額、残額、ステータス、催促履歴を表示
Payments API 一覧・詳細・登録・割当 入金情報と請求書への割当行を管理
Members API プロフィール・一覧・詳細・更新・無効化 会員自身の登録情報確認と、管理者による会員管理
Sales / Collections API 売上・顧客別集計・CSV・催促管理 請求ベースの売上、回収率、未回収、催促文面・履歴を表示

学び・今後の改善予定

学び

  • 同一のASP.NET Core APIに対してNext.jsとWPFという異なるクライアントを接続し、業務ロジックを再利用する構成
  • 一覧性の高いDataGrid、検索・ページング、ロール別導線を業務向けデスクトップ画面へ落とし込む設計
  • 画面状態・入力値・コマンド・非同期API通信をViewModelへ整理し、UIコードとの責務を分離する考え方
  • 入金割当、請求ステータス、売上・回収率など、請求・入金ドメインの状態を画面表示へ反映する実装
  • Fake Serviceを用いて外部APIから切り離し、ViewModelの検索条件・入力検証・DTO生成・エラー状態をテストする方法

今後の改善予定

  • 共通スタイルと再利用可能なUIコンポーネントの整理
  • 入力バリデーション、エラーハンドリング、認証切れ時の導線強化
  • Service／Dialog／ファイル出力の抽象化とテスト容易性の向上
  • 大量データを想定したDataGrid表示・検索・ページングの最適化
  • Web版・WPF版・共通APIの仕様差分をドキュメントで継続管理

※仮想センサー検証ではなく、請求・入金という業務ドメインを題材に統一し、WPF実務経験とASP.NET Core API活用の両方を示す構成にしています。
