小規模チーム向け 案件・タスク管理アプリ

状態遷移ルールを備えた、案件・タスク管理アプリです。
案件一覧、タスク一覧、詳細、作成、CSV出力など、業務アプリで必要になりやすい基本機能を備えつつ、 同一の業務ルールを Web / デスクトップの複数UIでどう表現するかを整理できる構成を目指しました。

技術構成：ASP.NET Core API / Next.js / Razor Pages / WPF

API GitHub Front GitHub Razor GitHub WPF GitHub

※ デモ環境は常時公開しておりません（コスト最適化のため）。
※ その代わり、GitHub・画面イメージ・構成説明で実装内容が確認できるようにしています。
※ 本作品は、業務ルールの設計意図と UI 構成の違いが伝わることを重視しています。

実装状況

API を共通基盤としつつ、UI ごとの実装方式の違いが分かるように段階的に整理しています。

実装済み（現時点）

  • 認証を前提としたユーザー判定・画面遷移
  • 案件一覧、タスク一覧、タスク詳細、タスク作成
  • 状態遷移ルールの検証（ToDo / Doing / Blocked / Done）
  • ロール別の操作制御（Leader / Worker）
  • 検索・絞り込み・ページング
  • CSV出力
  • Next.js / Razor Pages / WPF それぞれでの画面実装
  • CI による build / test の自動実行

※ UI 単体ではなく、業務ルールと画面導線の整合が確認できるものを実装済みとして扱っています。

これから（改善予定）

  • 状態遷移ルールの追加パターン対応
  • 権限差分に応じた UI 表示の整理
  • エラーハンドリング / メッセージ設計の強化
  • Integration Test / PageModel Test の拡充
  • README / 構成図 / 画面設計説明の拡充

「どこまで実装済みで、どこから改善予定か」を明確にし、要件整理の過程が伝わるようにしています。

設計方針

  • 業務ルールを UI 依存ではなく仕様として整理
  • API を共通化し、複数UIへ展開しやすい構成へ
  • 一覧・詳細・更新・CSVなど業務アプリの基本導線を重視

品質・テスト

  • CI（GitHub Actions）で build / test を自動実行
  • 重要ロジックの検証を優先してテスト実装
  • UI層ごとの差異を比較しやすいよう構成を整理

単なる画面実装ではなく、仕様変更や追加実装に耐えやすい土台づくりを意識しています。

構成・アーキテクチャ

API

ASP.NET Core API を共通基盤とし、認証、業務ルール、状態遷移検証、データアクセスを担当します。 UI に依存しないバックエンドとして整理しています。

Next.js Frontend

API 分離構成を前提にした Web フロントです。 一覧・詳細・状態更新などを SPA 寄りの画面構成で表現しています。

Razor Pages

ASP.NET Core 標準構成に近い UI 実装です。 PageModel ベースで、入力・検証・画面遷移・API 連携を整理しています。

WPF

デスクトップ業務アプリとしての操作性を意識した UI 実装です。 Windows 業務アプリ経験と接続しやすい構成で再整理しています。

※ 同一ドメインを複数UIで展開することで、責務分離・再利用性・表現差分を比較しやすくしています。

主な機能

状態遷移管理

  • ToDo / Doing / Blocked / Done の状態管理
  • 不正な遷移の抑止
  • 業務ルールとしての遷移制御

ロール制御

  • Leader / Worker の操作差分
  • 認証状態に応じた画面遷移
  • 権限に応じた更新制限

業務画面

  • 案件一覧 / タスク一覧 / タスク詳細
  • 新規作成 / 編集 / 絞り込み
  • CSV出力

継続改善

  • テスト追加
  • README / ドキュメント整備
  • UI 比較を通じた改善整理

学び・今後の改善予定

学び

  • 状態遷移ルールを単なる画面制御ではなく、業務上の制約として整理する重要性
  • 同一ドメインでも Next.js / Razor / WPF で関心事や実装の重心が変わること
  • API を共通基盤にすることで、UI 差し替えや比較がしやすくなること
  • 一覧・詳細・更新・CSV出力など、業務アプリでよく使う導線を共通観点として整理できたこと
  • 継続的な改善のために README・テスト・CI を土台として整える重要性

今後の改善予定

  • 状態遷移パターンやバリデーション条件の追加
  • テスト拡充（サービス層 / PageModel / Integration Test）
  • エラーページやログ出力の整理
  • 画面設計書・構成図を意識した説明資料の強化
  • UI 間比較をより明確にするための README 改善

※ 本作品では、機能の量よりも「業務ルールをどう設計し、どう UI に落とし込むか」を重視しています。
