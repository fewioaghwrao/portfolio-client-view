請求書・入金ステータスダッシュボード（Lite）

請求書・入金・ステータス管理を中心に、業務アプリとして必要になりやすい「一覧 / 検索 / 詳細 / 更新」を一通り揃えたダッシュボードです。
Lite版として要件を絞りつつも、設計（データ構造・画面遷移・API）と実装の整合が伝わる構成を目指しています。

デモを見る GitHub リポジトリ

※フロントエンドは Azure Static Web Apps、バックエンド API とデータベースは Heroku 上で公開しています。
※デモデータはダミーです。

実装状況

業務アプリとして必要なコア機能は一通り実装済みとし、 現在は運用・拡張を見据えた改善点を整理しています。

実装済み（現時点）

  • 認証/認可（ログイン・ロール制御）
  • 請求書：一覧/検索/詳細/作成/更新
  • 請求明細（InvoiceLines）：追加/更新/削除
  • 入金：一覧/検索/詳細/登録
  • ステータス（未入金／一部入金／入金済み）の自動判定
  • 催促（リマインド）：送信履歴管理
  • 売上集計：月次 / 顧客別 / ステータス別
  • 請求書PDF出力（管理者/会員）※QuestPDF + 日本語フォント埋め込み
  • 操作ログ：請求・入金・催促などの監査ログ記録および管理者向け一覧画面
  • 催促処理：非同期ジョブとして実行
  • 単体テスト：主要サービス層の業務ロジックを検証
  • 統合テスト：認証・認可・主要APIエンドポイントを検証
  • Docker / GitHub Actions によるビルド・テスト自動化

※機能は「動作する状態」を基準に記載しています（UIのみ/未結線は実装済みに含めません）。

これから（実装予定）

  • 月次締め処理（締め後の更新制御）
  • 催促テンプレートの管理機能
  • 監査ログ検索条件・CSV出力の拡充
  • 境界値・異常系テストの追加
  • 一覧・集計処理のパフォーマンス改善

「未実装」を明確にすることで、要件整理と優先順位付けの判断が伝わるようにしています。

設計方針

  • 関心の分離：API / Application / Domain / Infrastructure
  • DTO で入出力を固定し、内部実装の変更に強い構造へ
  • 一覧は標準で：検索条件・ページング・ソート

品質・テスト

  • CI（GitHub Actions）で build / test を自動実行
  • 入金割当・ステータス再計算など重要ロジックの単体テストを実装
  • 管理者 / 会員の認可について Integration Test で 403 を検証

業務上の重要度が高い処理から優先的にテストし、 仕様変更やリファクタリング時の安全性を確保しています。

画面キャプチャ

業務アプリを想定し、一覧・検索・更新の導線と、ロール制御（管理者/会員）を重視しています。

学び・今後の改善予定

学び

  • 業務ドメイン（請求・入金・一部入金）の状態遷移を、データ構造と画面導線に落とし込む設計
  • フロント（Next.js）↔ バック（ASP.NET Core）を REST API + JWT で疎結合にし、役割分担できる構成
  • 集計（売上/未入金/回収率）を「一覧・検索」と同じUXで扱うための API 設計（条件・ページング・サマリー）
  • “動くデモ”として成立させるためのダミーデータ設計と公開時の安全設計（権限/入力/ログ）
  • 認可（管理者/会員）や重要ロジックを CI 上の自動テストで継続的に検証し、リファクタリング耐性を確保

今後の改善予定

  • テスト拡充：重要ロジック（入金割当／ステータス再計算／集計）を中心に、 既存テストに網羅ケース（境界値・期限超過・複数割当）を段階的に追加
  • 運用強化：監査ログ（重要操作）／例外ハンドリングの統一／ログの相関ID付与
  • 業務拡張：月次締め（締め後の更新制御）／催促テンプレの管理（文面・差分履歴）
  • パフォーマンス：一覧の検索最適化（index設計）／集計のキャッシュ・非同期化（必要時）

※ Lite 版では「実務で重要度が高いコア機能（請求・入金・集計）」を優先し、 未実装の月次締めや催促テンプレート管理を中心に、運用機能を段階的に追加する方針です。
